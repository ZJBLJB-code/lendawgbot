# Red-Team Audit — Canonical Publisher Pivot (2026-04-27)

Adversarial audit of `tools/publisher.py`, `tools/publisher_schema.py`, and the
new `--from canonical` CI path before the architecture pivot lands.

## Summary

| Severity | Count | Status |
|---|---|---|
| P0 | 0 | — |
| P1 | 2 | Fixed inline (this PR) |
| P2 | 3 | Documented backlog |
| P3 | 4 | Documented backlog |

All P0/P1 issues are remediated in this commit. P2/P3 are tracked below for
the next iteration; none of them are exploitable in the current threat model.

## P1 — Fixed inline

### P1.1 — Corrupt / mid-write verdict silently emits zero-gates dashboard

**What:** When `pick_canonical_verdict()` returned the freshest canonical-named
verdict, `_build_day0()` called `_read_json()` which returned `None` on
JSONDecodeError, and the builder fell through to `raw = {} or {}`. The
resulting dashboard had `all_pass=False, gates_passed=0` and the headline
"Len Dawg is still being tested." A mid-write race with quant-bot, a
truncated network copy, or a literal `echo > verdict_*.json` flips the live
site from PASS to FAIL silently.

**Why it matters:** Reverses weeks of green-gate work invisibly. Operator
has no signal until they look at the dashboard. There was a parseable
variant on disk that would have worked.

**Fix:** Two-layer defence in `tools/publisher.py`:

1. `pick_canonical_verdict()` now validates each candidate via
   `_is_parseable_verdict()` (parses JSON + checks `gate` block exists)
   and skips corrupt files. Falls through canonical → variants in order.
2. `_build_day0()` raises `PublisherError` if its handed-in path is
   un-readable or missing `gate`. The picker shouldn't hand it a corrupt
   file anymore, but defense in depth.

Regression tests added: `test_corrupt_canonical_falls_through_to_valid_variant`
and `test_all_corrupt_verdicts_raises_publisher_error`.

### P1.2 — Unbounded JSONL line read (potential OOM)

**What:** `_read_jsonl()` and `_has_real_trade_event()` iterated `for line in
fh:` with no per-line size cap. A single 100 MB line (no newline) would
buffer the whole thing into memory before `json.loads` rejected it. A truly
adversarial 4 GB single-line file would page-fault the publisher.

**Why it matters:** Local-only input from a trusted bot, but a runaway log
or filesystem corruption would still take the publisher down — and the
publisher runs on Zach's laptop and in CI. A crashed publisher in CI means
the dashboard goes stale silently.

**Fix:** Added `MAX_JSONL_LINE_BYTES = 1 MiB` and `MAX_JSONL_LINES = 200_000`
caps. Lines over the byte cap are skipped with a warning; the iterator
breaks out at the line cap. quant-bot events are O(KB) so legitimate
traffic is nowhere near the cap.

## P2 — Backlog

### P2.1 — Path traversal in `--quant-bot-root`

`python3 tools/publisher.py --quant-bot-root /tmp/../../etc` doesn't blow up,
but it also doesn't sanitize. Today it's harmless because the publisher only
reads under `<root>/journal/...`, but if a future reader ever does
`open(root + user_supplied)` it could escape. The CLI is operator-controlled
in our threat model — not exploitable today. Backlog: add a
`Path(...).resolve()` and reject if no `journal/` subtree exists.

### P2.2 — Schema drift detection is one-way

The publisher accepts `gate.all_pass` as bool OR `"true"` OR truthy ints
(via `_b()`). That's defensive. But if quant-bot one day flips
`gate.in_sample_sharpe` to be a string `">=0.6"` instead of `1.02`, `_f()`
silently returns `None` and the gate value disappears from the dashboard —
no error, no warning. Backlog: emit warnings when a numeric coercion fails
on a field that the schema declares non-nullable.

### P2.3 — Atomic write on NFS

POSIX `rename()` is atomic on the same filesystem. `tempfile.mkstemp(dir=
out_path.parent)` keeps temp on the same FS as destination. Behavior is
correct on macOS (APFS) and Linux (ext4/xfs). NFSv3+ implements rename
atomically when source/dest live on the same export — the publisher does
this. SMB/CIFS shares MAY NOT — but neither host runs over SMB. Backlog:
document the FS assumption in the publisher header; consider an
`fsync(parent_dir)` for full crash-safety.

## P3 — Acknowledged

### P3.1 — `cumulative.current_equity` is `STARTING_EQUITY_DEFAULT (10_000) +
total_pnl`

Not real account equity. Floor is hardcoded constant; only the *delta* (sum
of per-trade pnl_dollars) reflects the bot. /MES @ 1 contract, total_pnl is
bounded ~$25-200/trade. **Not PII** — it's a synthetic display number, by
design. Document explicitly so a future code reader doesn't "fix" it back to
real equity.

### P3.2 — `Provenance.source_detail` echoes filenames

Today `verdict_2026-04-25.json`. Tomorrow `live/2026-04-28.jsonl`. Filenames
are not PII; they're literally what we want operators to see. Acknowledge.

### P3.3 — `_provenance.publisher_run_id` is a fresh UUID each build

This is desirable (lets us tell two builds apart), not a leak. But note: if
we ever switch to a deterministic seed for testing, the field needs to stay
unique-per-build in production. Acknowledge.

### P3.4 — Renderer XSS surface is `esc()` everywhere — confirmed

Every interpolation point in `src/template.html` that reaches innerHTML
passes through `esc()` (HTML-escape `&<>"'`) or `num()` (numeric coercion).
The data is delivered via `<script id="tomcash-data" type="application/json">`
and parsed with `JSON.parse(...textContent)` — never eval'd. The build step
also escapes `<`, `>`, `&`, U+2028, U+2029 in the embedded JSON to prevent
`</script>` breakout. Coverage is complete. **No fix needed.**

## Threat-model notes (informational)

- **Trust boundary:** quant-bot's filesystem is fully trusted (Zach's laptop).
  The publisher's job is not to defend against a malicious quant-bot; it's
  to ensure that if quant-bot ever **accidentally** emits a field with PII
  (account_id, equity_usd, etc.), the allow-listed schema strips it before
  it reaches the public dashboard.
- **Allow-list IS the contract:** `tools/publisher_schema.py` declares every
  field that can leave this process. New PII fields added to quant-bot in
  the future cannot survive the trip into `data/dashboard.json` because no
  field in the schema accepts them. This is the architectural inversion vs
  the legacy `scripts/sanitize_jsonl.py` (a blocklist that fails open on
  novel field names).
- **PII test re-run on real data (2026-04-27):** `data/dashboard.json`
  produced by the publisher against `~/quant-bot` was grep'd for the full
  PII_FIELDS list (`U25438416`, `DUP187165`, `equity_usd`, `account_id`,
  `client_id`, `host`, `port`, `bot_pid`, plus the 17 risk/cost knobs).
  **Zero hits.** The test in `tests/test_publisher.py::test_pii_never_leaves`
  also passes.

## Operational gotchas (informational, not findings)

- **Fresh quant-bot install (no day0 verdicts, no parquet, no LEARNINGS.md):**
  `detect_mode` raises `PublisherError("no usable artifacts")`. CLI returns 1.
  `sync_quant_bot.sh` exits 2. `refresh.sh` exits 1. CI's
  `if [ -f data/dashboard.json ]` guard then falls back to `--from sample`,
  so the public site stays green during cold-start. Deliberate.
- **Race with quant-bot mid-writing a verdict:** P1.1 fix above plus the
  caller's `os.replace` atomic-write means worst case is the previous
  good `data/dashboard.json` stays in place. No partial writes possible.
- **Operator path to root cause:** publisher logs at INFO/WARN/ERROR with
  module name. `sync_quant_bot.sh` reports schema_version + mode +
  source_detail + verdict line. `refresh.sh` tails `/tmp/lendawg-build.log`
  on failure and prints `_status.json` warnings/errors. Three layers, each
  one terser than the last — operator should always know which layer
  exploded.
